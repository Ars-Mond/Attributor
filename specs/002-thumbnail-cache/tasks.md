# Tasks: Photo Thumbnail Cache

**Input**: Design documents from `specs/002-thumbnail-cache/`

**Prerequisites**: plan.md, spec.md, research.md, data-model.md, contracts/

**Tests**: Included — the project has an integration-test suite and `quickstart.md` enumerates
thumbnail validation scenarios. Test inputs (landscape/portrait, JPEG/PNG/WebP) are generated
in-test via the `image` crate; existing fixtures under `src-tauri/test_images/` are also used.

**Organization**: Grouped by user story — US1 (viewer high preview), US2 (tree low previews),
US3 (cache reuse & locality). The generation engine and scan-time trigger are Foundational.

## Format: `[ID] [P?] [Story?] Description`

- **[P]**: Can run in parallel (different file, no dependency on an incomplete task)
- **[Story]**: US1 / US2 / US3 (user-story phases only)

## Generation model (per analyze remediation)

Thumbnails are generated by the backend **during the folder scan** (`scan_dir`): each image file
triggers `photo::thumbnail::ensure_thumbnails`, both variants are produced, and their paths are
written into `FileNode` (`thumb_low` / `thumb_high`). The file tree and viewer read these paths
directly. Generation is sequential within the scan (which already runs off the UI thread via
`spawn_blocking`); **no semaphore/rayon** is added to the photo module — bounding generation for
deep/recursive trees is **deferred to a future feature** (accepted tradeoff: folder open is slower).
A thin `get_thumbnails` command is kept only as a **viewer fallback** for files opened outside a scan.

## Path Conventions

Backend Rust under `src-tauri/src/`; integration tests under `src-tauri/tests/`; frontend under
`src/lib/` and `src/routes/`. Builds on the existing `photo` module and the `image` crate (no new deps).

## Phase 5: User Story 3 - Cache reuse and locality (Priority: P3)

**Goal**: Valid thumbnails are reused across sessions; invalid ones regenerate; failures degrade gracefully; `_thumbnail` stays out of the tree.

**Independent Test**: Generate, restart, rescan → no new files written; a 0-byte thumbnail regenerates; a corrupt source or read-only `_thumbnail` yields an error without crashing; a `_thumbnail` folder never appears in the tree.

- [ ] T011 [P] [US3] Add tests: reuse (2nd `ensure_thumbnails` regenerates nothing), regenerate-on-invalid (0-byte file), **graceful failure (corrupt source → `Err`, read-only `_thumbnail` dir → `Err`, no panic)**, and `_thumbnail` excluded from the `FileNode` tree — in `src-tauri/tests/thumbnail_test.rs`
- [ ] T012 [US3] Harden `is_valid` to reject empty/undecodable thumbnail files so invalid ones regenerate (FR-011) in `src-tauri/src/photo/thumbnail.rs`

**Checkpoint**: All three user stories are independently functional.

## Dependencies & Execution Order

- **Setup (T001)** → **Foundational (T002–T005)** → user stories.
- Within `thumbnail.rs`: T002 → T003 → T012 (same file, sequential). T004 (scan integration) depends on T003 (`ensure_thumbnails`); T005 depends on T003.
- **US1 (T006–T008)** and **US2 (T009–T010)** depend on Foundational; they touch different files (viewer/page vs tree) and can run in parallel — both read `FileNode` paths produced by T004.
- **US3 (T011–T012)** depends on Foundational; T012 edits `thumbnail.rs` (after T003).
- **Polish (T013–T015)** last; T015 after all implementation.

## Parallel Opportunities

- Each story's test task is parallel to that story's implementation (different files): T006 ∥ T007–T008; T009 ∥ T010; T011 ∥ T012.
- US1 and US2 implementation can run concurrently.
- Polish: T013 (Rust) ∥ T014 (frontend).
- Note: test tasks T006/T009/T011 share `thumbnail_test.rs` → sequential among themselves (different phases anyway).

## Implementation Strategy

- **MVP** = Setup + Foundational + **User Story 1** (viewer high preview). Ship/validate, then add US2 (tree), then US3 (reuse/error hardening).
- **Incremental**: the shared engine and scan-time trigger land in Foundational, so US1/US2 only wire UI to `FileNode` paths.

## Deferred (out of scope for this feature)

- Bounding/throttling generation for deep recursive trees (the scan generates all thumbnails up front, slowing folder open). To be addressed in a future feature; no semaphore/rayon is added to the photo module now.
